# AGENTS.md

## Назначение

Этот файл фиксирует правила работы AI-агентов и разработчиков в репозитории **AeroChat**.

Цель документа:
- защитить архитектурный фундамент проекта;
- сделать вклад в проект воспроизводимым;
- снизить риск деградации структуры репозитория;
- обеспечить безопасную и последовательную LLM-assisted разработку.

Этот документ обязателен к соблюдению для любых автоматизированных и ручных изменений в репозитории.

---

## 1. Общие принципы

### 1.1. Не ломать фундамент

Любое изменение должно:
- сохранять архитектурные границы;
- не ломать локальный запуск;
- не ломать CI;
- не ломать существующие контракты без явного решения через ADR;
- не вносить скрытую сложность.

### 1.2. Один PR — одна задача

Каждое изменение должно быть:
- небольшим;
- изолированным;
- понятным;
- проверяемым;
- завершённым.

Недопустимо:
- объединять в одном PR infra + auth + crypto + frontend + rtc;
- делать массовый рефакторинг без причины;
- менять архитектуру “заодно”.

### 1.3. Документация идёт вместе с кодом

Если изменение:
- вводит новый модуль;
- меняет контракты;
- меняет поведение системы;
- меняет ограничения;
- меняет flow разработки,

то оно должно обновлять:
- соответствующий ADR, если решение архитектурно значимо;
- README или docs, если меняется usage/dev flow;
- комментарии в коде, если без них теряется контекст.

### 1.4. Без симуляции безопасности

Недопустимо:
- писать “псевдо-E2EE”;
- добавлять crypto placeholder, который выглядит как готовая защита;
- внедрять небезопасные заглушки, замаскированные под production code;
- хранить секреты в коде;
- логировать чувствительные данные.

### 1.5. Русский язык документации

Для проекта зафиксировано:
- документация — **на русском языке**;
- комментарии в коде — **на русском языке**, если комментарий действительно нужен;
- commit messages — по формату Conventional Commits, на английском или русском, но кратко и по делу;
- названия сущностей и имена пакетов/модулей — на английском.

## 2. Обязательный workflow для разработки

### 2.1. Работа только через ветки

Запрещено работать напрямую в `main`.

Для любой задачи создаётся отдельная ветка.

Рекомендуемые шаблоны названий веток:

- `feat/<scope>-<short-name>`
- `fix/<scope>-<short-name>`
- `refactor/<scope>-<short-name>`
- `docs/<scope>-<short-name>`
- `chore/<scope>-<short-name>`
- `test/<scope>-<short-name>`
- `infra/<scope>-<short-name>`

Примеры:
- `feat/identity-service-bootstrap`
- `docs/adr-foundation`
- `infra/dev-compose-stack`
- `test/chat-service-unit-tests`

### 2.2. Перед началом работы

Перед любыми изменениями агент обязан:

1. Прочитать:
   - `README.md`
   - `AGENTS.md`
   - relevant ADR
   - затрагиваемые файлы

2. Проверить:
   - нет ли уже существующего решения;
   - не противоречит ли задача ADR;
   - не затрагивает ли изменение опасные домены.

3. Сформулировать план:
   - что меняется;
   - какие файлы меняются;
   - какие тесты будут добавлены или обновлены;
   - как будет проверяться результат.

### 2.3. После завершения работы

Перед завершением агент обязан:

- прогнать релевантные тесты;
- проверить линтеры;
- проверить форматирование;
- обновить документацию;
- убедиться, что нет мусорных файлов;
- убедиться, что в diff нет случайных изменений.

## 3. Правила коммитов

### 3.1. Использовать Conventional Commits

Формат:

```text
<type>(<scope>): <description>
````

Где:

* `type` — тип изменения;
* `scope` — модуль, сервис или область;
* `description` — краткое действие.

Примеры:

* `feat(identity): add service bootstrap`
* `fix(web): correct shell route fallback`
* `docs(adr): add storage and relay decision`
* `test(chat): add repository unit tests`
* `chore(repo): configure golangci-lint`

### 3.2. Допустимые типы

* `feat`
* `fix`
* `refactor`
* `docs`
* `test`
* `chore`
* `build`
* `ci`
* `perf`
* `revert`

### 3.3. Правила для commit message

Коммит должен быть:

* конкретным;
* коротким;
* без шума;
* без “misc”, “update stuff”, “changes”;
* без описания нескольких несвязанных изменений.

Плохо:

* `fix: updates`
* `feat: many improvements`
* `chore: changes`

Хорошо:

* `feat(gateway): add health and readiness endpoints`
* `docs(readme): describe local bootstrap flow`
* `test(identity): add config validation tests`

### 3.4. Большие задачи

Если задача большая, её нужно разбить на несколько коммитов и при необходимости на несколько PR.

## 4. Правила по архитектуре

### 4.1. Архитектурные инварианты

Следующие правила нельзя нарушать без нового ADR:

1. Проект остаётся **монорепозиторием**.
2. Backend остаётся **Go-first**.
3. Контракты между сервисами — **proto-first**.
4. Криптография не импровизируется.
5. E2EE не считается реализованной без отдельной утверждённой спецификации.
6. Медиа-контент не должен становиться постоянным plaintext storage на сервере.
7. Frontend shell и backend domains должны развиваться итерационно и независимо.
8. В одном PR нельзя одновременно перестраивать crypto, auth, storage и rtc.

### 4.2. Когда нужен новый ADR

Новый ADR обязателен, если изменение:

* меняет границы сервисов;
* меняет storage model;
* меняет auth model;
* меняет crypto model;
* меняет transport model;
* меняет deployment model;
* добавляет фундаментальную внешнюю зависимость;
* меняет стратегию self-host;
* меняет принципы UI shell.

### 4.3. Формат ADR

Каждый ADR должен содержать:

* заголовок;
* статус;
* контекст;
* решение;
* последствия;
* альтернативы при необходимости.

## 5. Правила по backend

### 5.1. Общие требования

Backend-код должен быть:

* простым;
* понятным;
* production-oriented;
* тестируемым;
* без магии;
* без ненужной абстракции.

### 5.2. Что обязательно

* контекстный `context.Context`;
* корректный graceful shutdown;
* структурированные логи;
* конфигурация через env/config layer;
* health/readiness endpoints;
* явные ошибки;
* unit/integration tests там, где это оправдано.

### 5.3. Что запрещено

* глобальные mutable singleton state;
* неявная магия DI;
* ручное копирование proto/generated code;
* ad-hoc SQL по проекту без общего правила;
* бизнес-логика в transport handlers;
* скрытая зависимость от конкретного контейнера/окружения;
* логирование токенов, паролей, ключей, plaintext payload’ов.

### 5.4. Комментарии в Go-коде

Комментарии писать только там, где без них теряется контекст:

* нестандартные решения;
* ограничения;
* инварианты;
* security-sensitive поведение;
* protocol-specific нюансы.

Не писать очевидные комментарии уровня:

* “создаём переменную”
* “проверяем ошибку”
* “возвращаем результат”

Все полезные комментарии — на русском языке.

---

## 6. Правила по frontend

### 6.1. Общие требования

Frontend должен:

* быстро загружаться;
* быть пригодным для слабых устройств;
* быть адаптивным;
* не быть перегруженным библиотеками;
* поддерживать дальнейшую PWA-эволюцию;
* сохранять визуальную идентичность AeroChat.

### 6.2. Инварианты

* Core shell должен быть лёгким.
* Heavy UI загружается lazy.
* Нельзя тянуть тяжёлую UI-библиотеку без очень веской причины.
* Нельзя внедрять raw HTML rendering для markdown.
* Нельзя ухудшать мобильный UX ради desktop-эстетики.

### 6.3. Производительность

Нужно стремиться к тому, чтобы:

* основной shell открывался быстро;
* переключение между окнами было отзывчивым;
* скролл чата не лагал;
* старые телефоны не умирали от декоративных эффектов.

### 6.4. Комментарии во frontend-коде

Те же правила:

* писать только полезные комментарии;
* комментарии на русском;
* не описывать очевидное.

## 7. Криптография и безопасность

### 7.1. Главный запрет

Агенту запрещено самостоятельно “придумывать” криптографический протокол.

### 7.2. Что можно делать

До утверждённой crypto implementation разрешено:

* проектировать интерфейсы;
* проектировать key storage abstraction;
* проектировать envelope format;
* писать тестовые заглушки;
* описывать ограничения;
* писать документацию;
* готовить integration points.

### 7.3. Что нельзя делать

Запрещено:

* объявлять задачу безопасной без оснований;
* писать самодельный шифр или handshake;
* использовать устаревшие или сомнительные примитивы;
* смешивать временные dev-заглушки с production semantics;
* хранить ключи в открытом виде в БД или логах.

### 7.4. Sensitive logging policy

Никогда не логировать:

* пароли;
* access/refresh tokens;
* ключевой материал;
* plaintext сообщений;
* plaintext вложений;
* полные секреты из env;
* приватные профилирующие данные.

---

## 8. Работа с тестами

### 8.1. Тесты обязательны

Если изменение меняет поведение системы, должны быть тесты.

Минимум:

* unit tests на доменную логику;
* integration tests на критичные инфраструктурные слои;
* smoke tests на основные сценарии, когда это уместно.

### 8.2. Coverage

Coverage не является самоцелью, но агент обязан:

* смотреть покрытие изменяемого модуля;
* не снижать покрытие без причины;
* добавлять тесты на новые ветви логики;
* особенно покрывать error paths и edge cases.

### 8.3. Что тестировать в первую очередь

Приоритет:

1. Конфигурация
2. Доменные правила
3. Репозитории / storage adapters
4. Транспортные контракты
5. Security-sensitive поведение
6. Регрессии

### 8.4. Что недопустимо

* фейковые тесты без meaningful assertions;
* snapshot tests без причины;
* игнорирование flaky тестов;
* скрытое пропускание тестов.

---

## 9. Работа с Docker, контейнерами и логами

### 9.1. Разрешения

Если у агента есть full access к окружению, он обязан использовать его аккуратно.

Разрешено:

* запускать контейнеры;
* смотреть логи контейнеров;
* проверять health;
* проверять сетевое взаимодействие локального dev stack;
* воспроизводить баги локально.

### 9.2. Обязанности при работе с контейнерами

Если изменение связано с infra или runtime, агент обязан:

* проверить `docker compose up`;
* посмотреть логи сервисов;
* проверить readiness/health;
* убедиться, что контейнеры не падают циклически;
* убедиться, что env-конфигурация описана.

### 9.3. Что запрещено

* коммитить локальные секреты;
* коммитить временные дампы;
* коммитить runtime-артефакты;
* оставлять сломанный compose stack.

---

## 10. Работа с интернетом и документацией

### 10.1. Проверка документации обязательна

Если агент:

* внедряет новый инструмент;
* меняет конфигурацию;
* меняет integration flow;
* добавляет внешнюю зависимость;

он обязан свериться с актуальной документацией.

### 10.2. Принцип

Не полагаться на память там, где может быть устаревшая информация.

### 10.3. Что фиксировать

Если изменение основано на внешней документации, агент должен:

* реализовать решение аккуратно;
* при необходимости обновить docs;
* избегать сомнительных и устаревших практик.

---

## 11. Правила по документации

### 11.1. Язык

Внутренняя документация проекта пишется на русском языке:

* README
* ADR
* docs
* комментарии
* инструкции для агентов

### 11.2. Стиль

Документация должна быть:

* конкретной;
* инженерной;
* без воды;
* без маркетинговой риторики;
* без противоречий.

### 11.3. Что обновлять

При изменении соответствующих областей обновлять:

* `README.md`
* `AGENTS.md`
* `docs/adr/*`
* `docs/roadmap.md`
* сервисные README при необходимости

---

## 12. Pull Request policy

### 12.1. Каждый PR должен содержать

* цель изменения;
* список основных изменений;
* риски;
* что было протестировано;
* что ещё не сделано;
* ссылку на ADR, если применимо.

### 12.2. PR checklist

Перед открытием PR агент должен убедиться, что:

* [ ] создана отдельная ветка
* [ ] изменение изолировано
* [ ] код отформатирован
* [ ] тесты добавлены или обновлены
* [ ] линтеры пройдены
* [ ] документация обновлена
* [ ] нет секретов и мусорных файлов
* [ ] diff понятен и не содержит случайных изменений

---

## 13. Ограничения по scope

Нельзя в одном изменении одновременно:

* внедрять auth flow;
* менять crypto model;
* добавлять chat storage semantics;
* добавлять rtc signaling;
* перестраивать frontend shell;
* переписывать infra.

Если задача затрагивает несколько доменов, она должна быть разбита.

---

## 14. Что считается хорошим результатом

Хорошее изменение:

* минимально по размеру;
* даёт реальную ценность;
* не ломает архитектуру;
* сопровождается тестами;
* сопровождается документацией;
* читается без догадок;
* оставляет проект в лучшем состоянии, чем до изменения.

---

## 15. Что считается плохим результатом

Плохое изменение:

* “вроде работает”;
* смешивает несколько задач;
* тащит лишние зависимости;
* добавляет security theater;
* не имеет тестов;
* не обновляет docs;
* не соблюдает границы сервисов;
* ухудшает DX;
* затрудняет дальнейшую работу людей и LLM.

---

## 16. Приоритеты проекта

При конфликте решений приоритеты такие:

1. Безопасность
2. Корректность
3. Простота архитектуры
4. Воспроизводимость
5. Производительность
6. DX
7. Визуальная полировка

---

## 17. Поведение AI-агента по умолчанию

Если агент не уверен:

* не додумывать архитектуру;
* не импровизировать в крипте;
* не объединять несвязанные задачи;
* сначала обновить docs/ADR;
* выбрать минимальное безопасное решение;
* явно отметить TODO только если он конкретный и полезный.

---

## 18. Финальное правило

Если изменение потенциально затрагивает фундамент проекта, но не отражено в документации, сначала меняется документация, потом код.
