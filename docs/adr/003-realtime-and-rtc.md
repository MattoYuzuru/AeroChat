# ADR-003: Realtime-коммуникация, signaling и RTC-границы

- Статус: Accepted
- Дата: 2026-03-16

## Контекст

AeroChat должен поддерживать:

- быстрый обмен сообщениями;
- realtime-события;
- typing/presence/read receipts;
- 1:1 звонки;
- групповые звонки;
- дальнейшее развитие в сторону screen sharing и recording.

Нужно заранее разделить:

- messaging realtime;
- call control plane;
- media plane.

Если этого не сделать, chat и RTC логика быстро смешаются в один трудно поддерживаемый слой.

## Решение

### 1. Разделение на planes

Проект разделяет коммуникацию на три слоя:

1. **Messaging realtime layer**
    - доставка realtime-событий;
    - presence;
    - typing;
    - receipts;
    - chat updates.

2. **RTC control plane**
    - signaling;
    - call lifecycle;
    - call permissions;
    - participant state;
    - room semantics;
    - recording metadata;
    - будущие политики screen share.

3. **Media plane**
    - передача audio/video/screen streams;
    - отдельный специализированный RTC/SFU слой.

### 2. Сервисные границы

За messaging realtime и API orchestration отвечает backend.

За call control отвечает `aero-rtc-control`.

Media plane допускается как отдельный специализированный компонент, а не как часть обычного chat-сервиса.

### 3. Групповые звонки

Групповые звонки проектируются через отдельный media-plane, совместимый с SFU-моделью.

### 4. Жизненный цикл группового звонка

На старте фиксируется следующая семантика:

- звонок в группе может переживать вход и выход участников;
- комната живёт, пока в ней есть хотя бы один участник;
- завершить звонок для всех может только главный управляющий субъект по политикам группы;
- пользовательские права в звонке определяются ролями и call policies.

### 5. Ограничения первого этапа

На фундаментальном этапе не реализуются:

- полная media pipeline;
- advanced recording;
- transcription;
- whiteboard;
- сложные screen sharing сценарии.

Но интерфейсы и доменные модели должны учитывать дальнейшее расширение.

## Последствия

### Положительные

- Chat и RTC не смешиваются.
- Появляется понятная точка роста для звонков.
- Упрощается тестирование control plane отдельно от media plane.

### Отрицательные

- Архитектура становится чуть сложнее с самого начала.
- Появляется дополнительный доменный сервис.

## Альтернативы

### 1. Встроить signaling и media semantics в chat service

Не выбрано, потому что это создаёт сильную связанность и быстро усложнит кодовую базу.

### 2. Делать звонки как побочную функцию frontend-only логики

Не выбрано, потому что call lifecycle, права и room semantics должны контролироваться серверной доменной моделью.

### 3. Реализовать только 1:1 звонки без задела на группы

Не выбрано, потому что продукт с самого начала ориентирован на группы и роли.
