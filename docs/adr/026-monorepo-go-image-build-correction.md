# ADR-026: Корректировка сборки Go-образов в монорепозитории для release workflow

- Статус: Accepted
- Дата: 2026-04-04

## Контекст

После внедрения GHCR-based image publish flow проект получил подтверждённый release blocker:
публикация Go-образов падает при сборке сервисов, которые используют monorepo-local Go module replacements.

Текущие сервисы `aero-gateway`, `aero-identity` и `aero-chat` зависят от внутренних модулей:

- `github.com/MattoYuzuru/AeroChat/gen/go`
- `github.com/MattoYuzuru/AeroChat/libs/go`

Эти зависимости подключаются через `replace` в сервисных `go.mod`, например:

- `../../gen/go`
- `../../libs/go`

Такая схема является допустимой для монорепозитория и соответствует уже принятым архитектурным ограничениям:

- проект остаётся монорепозиторием;
- внутренние Go-модули не публикуются отдельно;
- release workflow должен собирать контейнеры прямо из текущего репозитория;
- local/dev flow не должен ломаться.

Проблема возникает, если Docker build stage не воспроизводит внутри контейнера тот же layout монорепозитория, на который
рассчитаны относительные `replace`.

Подтверждённый пример: `aero-identity` использует `../../gen/go`, но его Dockerfile не копировал `gen/go` в build stage,
из-за чего `go build` внутри publish workflow завершался ошибкой вида `replacement directory ../../gen/go does not exist`.

Нужно исправить это минимально, не меняя модульную структуру репозитория и не вводя отдельную публикацию внутренних
Go-модулей.

## Решение

### 1. Release image build для Go-сервисов сохраняет monorepo-root context

Сборка release-образов для `aero-gateway`, `aero-identity` и `aero-chat` продолжает выполняться из корня репозитория.

Это означает, что Dockerfile сервиса обязан явно копировать в build stage все внутренние модули и исходники,
необходимые для разрешения локальных `replace`.

### 2. Docker build stage обязан воспроизводить ожидаемый layout монорепозитория

Внутри build stage должны существовать пути, совместимые с сервисными `go.mod`:

- `/src/services/<service>`
- `/src/gen/go`
- `/src/libs/go`

Если сервис использует локальный `replace` на внутренний модуль, соответствующий каталог обязан быть скопирован в тот
же относительный путь.

### 3. Внутренние Go-модули не публикуются отдельно

`gen/go` и `libs/go` остаются monorepo-local модулями.

В рамках этого исправления запрещается:

- переносить их в отдельные registry/module releases;
- удалять `replace` ради container build;
- перестраивать сервисные границы или Go module layout.

### 4. Publish model не меняется

GitHub Actions publish workflow продолжает:

- собирать отдельные образы сервисов;
- публиковать их в GHCR;
- использовать текущую tag-модель `edge`, `sha-*` и `vX.Y.Z`;
- сохранять web image publish без изменения контракта.

Изменение относится только к корректности Docker build inputs для Go-сервисов.

### 5. CI получает явную проверку release image build

Чтобы регрессия ловилась до публикации образов, CI дополнительно валидирует сборку release-образов для:

- `web`
- `aero-gateway`
- `aero-identity`
- `aero-chat`

Эта проверка не публикует образы и не меняет deploy topology.
Она лишь подтверждает, что текущий репозиторий можно собрать в release-compatible контейнеры.

## Последствия

### Положительные

- Publish workflow снова совместим с monorepo-local `replace`.
- Внутренние Go-модули продолжают жить внутри монорепозитория без отдельной публикации.
- Release image model из ADR-020 сохраняется без topology drift.
- CI начинает ловить поломку image build до выхода в publish pipeline.

### Отрицательные

- Dockerfile сервисов остаются завязаны на явный monorepo layout внутри build stage.
- При появлении новых внутренних модулей нужно будет так же явно следить за их наличием в container build context.

### Ограничения

- Нельзя менять это исправление в redesign Go module structure без нового ADR.
- Нельзя переводить publish workflow на сборку из service-only context, если сервис использует monorepo-local `replace`.
- Нельзя считать CI image build заменой полноценного publish или deploy smoke.

## Альтернативы

### 1. Публиковать `gen/go` и `libs/go` как отдельные внутренние модули

Не выбрано, потому что это усложняет release model и не требуется для текущего монорепозитория.

### 2. Убрать `replace` и копировать generated/shared code в каждый сервис

Не выбрано, потому что это создаёт дублирование и ухудшает сопровождаемость.

### 3. Собирать каждый сервис только из его локального каталога

Не выбрано, потому что такая модель несовместима с уже принятыми monorepo-local зависимостями без дополнительного
усложнения.
