# ADR-001: Базовая архитектура AeroChat

- Статус: Accepted
- Дата: 2026-03-16

## Контекст

AeroChat создаётся как self-hosted чат-приложение с акцентом на:

- приватность;
- высокую отзывчивость;
- удобный desktop-like интерфейс;
- поддержку мобильных устройств;
- дальнейшее развитие в сторону E2EE, RTC и PWA.

Проект должен:
- быть воспроизводимым;
- хорошо подходить для публичного монорепозитория;
- позволять безопасно использовать AI-ассистентов в разработке;
- не разрастись в неуправляемый набор хаотичных сервисов.

Также важно:
- использовать Go как основной backend-язык;
- сохранить понятные bounded contexts;
- не перегрузить первый этап избыточной микросервисностью;
- подготовить платформу для будущих real-time и media фич.

## Решение

Принято решение использовать следующую базовую архитектуру:

### 1. Формат репозитория

Проект ведётся как **монорепозиторий**.

В монорепозитории размещаются:
- frontend-приложение;
- backend-сервисы;
- общие библиотеки;
- protobuf-контракты;
- документация;
- dev/prod infrastructure-конфигурация.

### 2. Архитектурный стиль

На старте используется **modular monolith mindset with service boundaries**, реализованный через несколько отдельных Go-сервисов в одном репозитории.

На первом фундаментальном этапе выделяются следующие домены:

- `aero-gateway`
- `aero-identity`
- `aero-chat`
- `aero-rtc-control`
- `aero-jobs`

Дополнительно допускаются:
- отдельный media-plane / SFU;
- отдельный временный encrypted relay для медиа.

### 3. Backend-подход

Backend остаётся **Go-first**.

Go используется для:
- внешних API;
- доменной логики;
- realtime orchestration;
- background jobs;
- инфраструктурных обвязок;
- конфигурации и observability.

### 4. Frontend-подход

Frontend реализуется как:
- веб-приложение;
- PWA-ready клиент;
- desktop-like shell на ПК;
- fullscreen shell на мобильных устройствах.

### 5. Контракты

Межсервисные и типизированные API-контракты проектируются как **proto-first**.

Protocol Buffers используются для:
- стабилизации контрактов;
- code generation;
- будущей межсервисной совместимости;
- упрощения безопасной итерационной разработки.

### 6. Хранение

На старте используются:
- PostgreSQL для основной модели данных;
- Redis для ephemeral state, rate limiting и short-lived coordination;
- MinIO для временного encrypted blob storage, если он нужен системе.

### 7. Деплой

Проект ориентируется на self-host deploy на один сервер с контейнерами.

Первичный сценарий:
- Docker Compose
- reverse proxy
- GHCR-based image delivery
- GitHub Actions для CI

## Последствия

### Положительные

- Репозиторий остаётся компактным и управляемым.
- Упрощается локальная разработка.
- Снижается операционная сложность на раннем этапе.
- Появляются чёткие границы доменов.
- Проект хорошо подходит для постепенного масштабирования.

### Отрицательные

- Некоторые части будут избыточно разделены для маленького проекта.
- Потребуется дисциплина, чтобы не нарушать сервисные границы внутри монорепо.
- На ранних этапах некоторые сервисы будут содержать минимум логики.

### Ограничения

- Нельзя хаотично плодить сервисы.
- Нельзя переносить бизнес-логику между доменами без причины.
- Нельзя нарушать инварианты архитектуры без нового ADR.

## Альтернативы

### 1. Один большой backend-сервис

Не выбрано, потому что это быстрее приведёт к смешиванию доменов и усложнит дальнейшее развитие.

### 2. Полноценная распределённая микросервисная система с первого дня

Не выбрано, потому что это слишком дорого операционно и организационно для текущего этапа.

### 3. Полностью P2P-архитектура

Не выбрано, потому что это плохо сочетается с удобством, офлайн-доставкой, multi-device и простым self-host.
