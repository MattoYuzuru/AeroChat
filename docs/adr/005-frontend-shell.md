# ADR-005: Frontend shell, desktop-like UX и мобильная адаптация

- Статус: Accepted
- Дата: 2026-03-16

## Контекст

Одна из отличительных черт AeroChat — визуальная идентичность.

Проект должен сочетать:

- эстетику Frutiger Aero / old desktop UI;
- необычный desktop-like shell на ПК;
- удобный fullscreen UX на мобильных устройствах;
- высокую производительность даже на старых устройствах.

Есть риск, что при попытке буквально эмулировать старую ОС:

- интерфейс станет тяжёлым;
- мобильный UX пострадает;
- декоративные элементы ухудшат отзывчивость.

Нужно зафиксировать продуктовый компромисс.

## Решение

### 1. Общий frontend-подход

Frontend строится как:

- современное веб-приложение;
- PWA-ready клиент;
- desktop-like shell на ПК;
- mobile-first practical adaptation на смартфонах.

### 2. Desktop UX

На ПК допускаются:

- иконки приложений;
- окна;
- перемещение окон;
- изменение размеров окон;
- полноэкранный режим окна;
- одновременная работа с несколькими чатами.

Но это остаётся UI-shell, а не буквальная эмуляция операционной системы.

### 3. Mobile UX

На мобильных устройствах:

- shell упрощается;
- рабочий стол может быть paged/scrollable;
- каждое “приложение” открывается полноэкранно;
- приоритет отдаётся удобству, а не точной симуляции desktop UX.

### 4. Performance-first правило

Декоративный стиль не должен ломать производительность.

Следствия:

- heavy modules загружаются lazy;
- core chat path остаётся лёгким;
- не допускается heavy UI framework без причины;
- анимации, blur и визуальные эффекты применяются умеренно;
- accessibility и responsiveness обязательны.

### 5. Markdown policy

Во frontend не допускается raw HTML rendering для пользовательских сообщений.

### 6. Drafts и локальное состояние

Frontend должен быть готов к:

- локальному кешу недописанных сообщений;
- восстановлению draft state;
- сохранению прикреплённых, но ещё не отправленных вложений;
- дальнейшему offline-friendly поведению.

## Последствия

### Положительные

- Сохраняется уникальность продукта.
- Не жертвуется мобильный UX.
- Производительность фиксируется как архитектурное требование, а не как пожелание.

### Отрицательные

- Desktop shell потребует отдельной дисциплины в реализации.
- Придётся постоянно балансировать между визуальной выразительностью и лёгкостью интерфейса.

## Альтернативы

### 1. Обычный стандартный chat UI без shell-концепции

Не выбрано, потому что это уничтожает основную эстетическую идентичность проекта.

### 2. Полная буквальная эмуляция desktop OS

Не выбрано, потому что это ухудшит usability и производительность, особенно на мобильных устройствах.

### 3. Одинаковый UI для ПК и телефона

Не выбрано, потому что это приведёт либо к неудобному desktop UX, либо к неудобному mobile UX.
