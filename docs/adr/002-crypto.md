# ADR-002: Криптографическая модель и границы ответственности сервера

- Статус: Accepted
- Дата: 2026-03-16

## Контекст

Одна из главных целей AeroChat — сделать так, чтобы владелец сервера не мог читать содержимое пользовательских
сообщений.

При этом продукт должен оставаться:

- удобным;
- быстрым;
- пригодным для self-host;
- совместимым с многопользовательскими чатами, вложениями и будущими звонками.

Нужно заранее зафиксировать важный принцип:
криптография в проекте не может проектироваться хаотично или “по вдохновению”.

Также нужно определить:

- что сервер может знать;
- что сервер не должен знать;
- как проектировать дальнейшие crypto-интерфейсы;
- что можно делать на старте, а что нельзя.

## Решение

### 1. Модель безопасности

AeroChat проектируется как **privacy-oriented encrypted messaging system**, где сервер:

- может хранить зашифрованные payload’ы;
- не должен иметь доступа к plaintext контенту сообщений;
- не должен иметь доступа к ключам расшифровки сообщений;
- не должен хранить ключевой материал в открытом виде.

### 2. Инвариант проекта

Сервер не должен считаться доверенной стороной для чтения пользовательской переписки.

### 3. Стратегия внедрения

Криптография внедряется **поэтапно**:

1. сначала проектируются identity и device model;
2. затем проектируются message envelopes и storage semantics;
3. затем фиксируется crypto spec;
4. затем внедряется реализация;
5. затем добавляются тесты и security review.

### 4. Что разрешено до полной реализации

До полноценной crypto-реализации разрешено:

- описывать интерфейсы;
- описывать device/session model;
- описывать envelope formats;
- описывать key storage abstraction;
- писать тестовые заглушки, которые явно помечены как незащищённые.

### 5. Что запрещено

Запрещено:

- объявлять систему E2EE-ready без реальной реализации;
- использовать самодельные криптопротоколы;
- смешивать dev-заглушки с production-семантикой;
- хранить plaintext ключи в БД;
- логировать ключевой материал;
- логировать plaintext сообщений.

### 6. Модель устройства и ключей

Проект должен быть готов к модели, в которой:

- у аккаунта есть набор устройств;
- у каждого устройства есть собственная криптографическая идентичность;
- ключи и сессии привязаны к устройствам;
- у пользователя есть статус резервного копирования ключей.

### 7. Вложения и медиа

Если система временно хранит вложения, они должны:

- рассматриваться как encrypted blobs;
- быть недоступны серверу в открытом виде;
- удаляться по TTL и политикам очистки.

## Последствия

### Положительные

- Снижается риск архитектурной деградации.
- Убирается соблазн “сделать псевдо-безопасность”.
- Создаётся основа для последовательного E2EE-внедрения.

### Отрицательные

- Реализация будет сложнее и дольше.
- Некоторые фичи нельзя будет быстро выпускать “в лоб”.
- Понадобится строгая дисциплина в моделировании identity, sessions и storage.

## Альтернативы

### 1. Доверять серверу и шифровать только storage-at-rest

Не выбрано, потому что это не решает ключевую продуктовую цель.

### 2. Делать E2EE “потом”, не фиксируя правила сейчас

Не выбрано, потому что это почти гарантированно приведёт к неправильным интерфейсам и миграционным проблемам.

### 3. Полностью отказаться от хранения чего-либо на сервере

Не выбрано, потому что это слишком сильно ухудшает UX, доставку, историю и multi-device.
